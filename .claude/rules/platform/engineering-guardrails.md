# Engineering Guardrails — What Makes Novara a Platform of the Future

## The Standard
Every line of code that enters Novara must make the platform BETTER, not just BIGGER.
Every release must solve a REAL customer problem, not add surface area.
Every decision must be defensible 5 years from now.

---

## Guardrail 1: No Code Without a Customer Problem

Before writing ANY code, answer: "Which customer problem does this solve?"

- If the answer is "it's cool technology" → don't build it
- If the answer is "the architecture needs it" → is the architecture serving customers or serving itself?
- If the answer is "customer X can't do Y because Z" → build it

**How this prevents what happened:**
The modularization was architecture serving itself. The customer didn't ask for modules — they asked for features that work. We should have asked: "Does modularization solve a customer problem TODAY?" The answer was no — it solves a developer scaling problem. That's valid, but it's a LOWER priority than making existing features work perfectly.

**Enforcement:** Every feature/task must have a `CustomerProblem` field. No field, no work.

---

## Guardrail 2: Working Software Over Perfect Architecture

A monolith that works perfectly for 100 customers is better than 17 modules where none are fully tested.

- Ship what works. Improve incrementally.
- Never break working software to achieve architectural goals.
- The monolith is not the enemy. Bugs are the enemy. Downtime is the enemy. Data loss is the enemy.
- Refactor UNDER the hood while the customer sees zero change.

**How this prevents what happened:**
We should have kept the monolith running and extracted ONE module completely, proven it worked identically, and only then moved to the next. Instead we touched everything and completed nothing.

**Enforcement:** Any architectural change must maintain backward compatibility. Old endpoints keep working. New endpoints are additive. Nothing breaks.

---

## Guardrail 3: Every Change Has a Before and After

Before committing ANY change, capture:
- BEFORE: What does the system do right now? (curl output, DB state, screenshot)
- AFTER: What does it do with my change? (same curl, same DB query, same screen)
- DIFF: What changed? Is that change intentional?

If you can't show before/after, you can't prove your change works.

**How this prevents what happened:**
We declared 54 endpoints "working" based on HTTP 200. We never compared the RESPONSE BODY field-by-field with the monolith. A 200 with wrong data is worse than a 500 — it silently corrupts the customer's understanding.

**Enforcement:** Every commit message or PR includes verification evidence. Not "tested locally" — actual curl output or screenshot diff.

---

## Guardrail 4: One Owner, Full Responsibility

Every module, every feature, every SP has ONE owner who is responsible for its correctness.

- "The agent generated it" is not ownership. YOU reviewed it, YOU tested it, YOU own it.
- If you didn't read every line, you don't own it. If you don't own it, don't commit it.
- Code without an owner decays. Code with an owner improves.

**How this prevents what happened:**
17 modules were generated by agents. Nobody read every line. Nobody owned the correctness of each SP call. The code had no owner, so nobody caught the 40+ mismatches.

**Enforcement:** Every file header says who owns it. Every PR has one reviewer who takes ownership of the change.

---

## Guardrail 5: Measure Twice, Cut Once

Before making an irreversible change:
1. Write down what you're about to do
2. Write down what could go wrong
3. Write down how you'd undo it
4. Then do it

Irreversible changes: deleting code, changing API contracts, modifying DB schemas, decommissioning services.

**How this prevents what happened:**
The plan said "Phase 5: Dissolve Monolith." If we had followed through, we would have deleted working software and replaced it with untested modules. The monolith was the safety net. We almost cut the safety net.

**Enforcement:** Any destructive operation requires explicit CTO approval AND a rollback plan documented BEFORE execution.

---

## Guardrail 6: Small, Verified, Incremental

The safest change is a small one. The most dangerous change is a big one.

- Add one endpoint, test it, ship it. Not 54 endpoints at once.
- Migrate one table, verify it, move on. Not 13 schemas at once.
- Extract one module, prove it works, then the next. Not 17 in parallel.

Big changes feel productive. Small verified changes ARE productive.

**How this prevents what happened:**
One session tried to extract 17 modules, test 54 endpoints, fix 40 SP mismatches, validate write operations, AND declare the architecture complete. The scope was impossible. If we had done Issues → verified → Artifacts → verified → Learn → verified, each module would have been correct.

**Enforcement:** PRs should be < 500 lines. If bigger, split them. Each PR must be independently testable and deployable.

---

## Guardrail 7: The Customer Doesn't See Architecture

Customers see: features that work, pages that load, data that's correct, actions that complete.
Customers don't see: module boundaries, SP names, SDK interfaces, gateway routing.

Every engineering decision should be evaluated by: "Does this make the customer's experience better?"

- Better performance → yes
- Better reliability → yes
- Better error messages → yes
- Cleaner module boundaries → no (unless it enables the above)
- More NuGet packages → no
- More repos → no

Architecture is a MEANS to serve customers. It is never the END.

---

## Guardrail 8: Respect the Code That Came Before

Every line of existing code represents a decision someone made, a bug someone fixed, a customer problem someone solved. Before changing it:

1. Understand WHY it exists
2. Understand WHAT breaks if you remove it
3. Understand WHO depends on it

If you can't answer all three, you're not ready to change it.

The IssueService's 81 lines weren't elegant. They were CORRECT. They called the right SPs with the right parameters. When we "improved" them in the modules, we broke them. The original was better.

---

## Guardrail 9: Errors Are Information, Not Embarrassment

When something breaks:
1. Don't hide it. Don't rush past it. Don't blame the tool.
2. Understand WHY it broke.
3. Fix the ROOT CAUSE, not the symptom.
4. Add a guardrail so it can't break the same way again.
5. Share the learning — someone else will make the same mistake.

When we found the SP mismatches, the right response was not "let me fix all 40 fast." It was: "Why did 40 mismatches happen? Because agents generated code without reading the source. The fix is: never let agents write service code. The guardrail is: the STOP rules."

---

## The Daily Practice

At the start of every session:
- What customer problem am I solving today?
- What is the smallest change that solves it?
- How will I prove it works?

At the end of every session:
- Did I leave the codebase BETTER than I found it?
- Can the next person continue my work without asking me?
- Would I be proud to show this to a customer?

If the answer to any of these is no — you're not done yet.
